# Аудит: 401 Unauthorized на /api/sessions/{id}/events при reload

Дата: 2026-07-21
Режим: read-only, код не менялся, коммитов нет.

## Executive summary

401 выдаёт **auth_guard middleware** для ВСЕХ запросов `/api/*` без `Authorization: Bearer` заголовка. Frontend подключается к events через нативный `new EventSource(url, { withCredentials: true })`, а **EventSource API не умеет задавать кастомные заголовки** — access token (localStorage) в запрос не попадает, а единственная auth-cookie (`refresh_token`) для API-авторизации не используется. Поэтому SSE-запрос получает 401 `{"detail":"missing_bearer"}` — при каждом подключении, включая каждый reload страницы. Проблема не специфична для reload: SSE не авторизовался никогда с момента введения фичи (#555).

## 1. Endpoint

- Backend: `backend/app/routers/session_events.py:67` — `GET /api/sessions/{session_id}/events` (SSE, `StreamingResponse`). Сама ручка проверяет доступ через `_legacy_load_session_scoped` → вернула бы 404 при отсутствии доступа. Но до неё запрос не доходит.
- Frontend: `frontend/src/hooks/useSessionEvents.js:77` — `new EventSource(url, { withCredentials: true })`; URL из `apiRoutes.sessions.events` (`frontend/src/lib/apiRoutes.js:165`).

## 2. Auth method

- Backend: JWT Bearer. `auth_guard_middleware` (`backend/app/startup/middleware.py:101-140`) для каждого `/api/*` пути, кроме `AUTH_PUBLIC_PATHS` (`backend/app/_legacy_main.py:232-...`), вызывает `user_from_bearer_header()` (`backend/app/auth.py:672`). Нет заголовка → `AuthError("missing_bearer")` → **401**. `/api/sessions/{id}/events` в `AUTH_PUBLIC_PATHS` отсутствует.
- Frontend: access token в localStorage (`fpc_auth_access_token`, `frontend/src/lib/apiCore.js:6`); обычные API-запросы добавляют `Authorization: Bearer <token>` (`apiClient.js:107-108`).
- Cookie: только `refresh_token` (HttpOnly, path=/, SameSite=Lax; `utils/auth_helpers.py`) — используется исключительно `/api/auth/refresh`, auth_guard её не читает.
- Token в SSE-запрос передаётся: **никак** (EventSource шлёт только cookies).

## 3. Экспериментальное подтверждение (stage + dev)

```
GET /api/sessions/test123/events  (без заголовков)
→ 401 {"detail":"missing_bearer"}        ← auth_guard

GET /api/sessions/test123/events  (Authorization: Bearer <jwt>)
→ 404 {"detail":"Not Found"}             ← auth прошёл, сработала логика ручки
```

Воспроизведено и на stage (`stage.processmap.ru`), и на dev-инстансе.

## 4. Гипотеза (подтверждённая)

**A: EventSource не передаёт Authorization header** — подтверждена кодом и экспериментом. Остальные гипотезы (SameSite cookie, CORS, JWT expiry, session store, отсутствие middleware на ручке) отвергнуты: cookie для API-auth не существует, CORS не причём (401 приходит JSON-ом от middleware), с валидным Bearer ручка работает.

«При reload» — просто самый заметный момент: EventSource переподключается при каждой загрузке страницы и каждый раз получает 401 (плюс браузерный auto-reconnect loop добавляет шум в консоль).

## 5. Рекомендации по фиксу

1. **Query-param токен для SSE (рекомендуется, low complexity)** — frontend: `eventsUrl()` добавляет `?access_token=<jwt>` из localStorage (token там уже есть). Backend: в `auth_guard_middleware` fallback — если заголовка нет, читать `request.query_params.get("access_token")` (глобально или только для `*/events`). Риск: токен в URL может попадать в access-логи — смягчается тем, что логирование query на наших proxy не настроено; можно ограничить fallback только SSE-путями.
2. **fetch-based SSE (frontend-only, medium)** — заменить нативный EventSource на fetch + ReadableStream (напр. `@microsoft/fetch-event-source`) с заголовком Authorization. Backend не трогается. Минусы: ручная реализация reconnect/backoff, чужая зависимость.
3. **Одноразовый SSE-ticket (high, самый безопасный)** — `POST /api/sessions/{id}/events/ticket` (Bearer) выдаёт короткоживущий одноразовый тикет, SSE подключается `?ticket=...`, middleware принимает тикет только для events-пути. Больше всего работы, но токен не утекает в логи/историю.

## 6. Приоритет

**Medium-low.** Real-time уведомления об удалении сессии (#555) фактически не работают с момента внедрения, но критичной функциональности это не ломает — приложение живёт без SSE. Однако: консольный шум от reconnect-loop и ложные 401 в мониторинге мешают. Вариант 1 закрывает проблему за ~30 строк.

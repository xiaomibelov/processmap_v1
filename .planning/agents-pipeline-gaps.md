# ProcessMap Agents Pipeline — Недостатки и пробелы

## CRITICAL — Блокируют результат

### 1. Нет автоматического деплоя после изменений кода
- **Проблема**: Agent 2 изменяет frontend/src/, собирает dist, но НЕ пересобирает Docker-образ gateway и НЕ перезапускает контейнер.
- **Последствие**: Пользователь не видит UI-изменений, потому что gateway (:5180) продолжает отдавать старый JS-бандл, встроенный при сборке образа.
- **Решение**: Добавить в EXECUTOR_PROMPT.md инструкцию: после npm run build выполнить docker compose -p processmap_test up -d --build gateway (или docker cp dist + nginx reload).

### 2. Нет runtime verification после выполнения
- **Проблема**: Нет автоматической проверки, что изменения работают (curl, screenshot, Playwright).
- **Последствие**: Agent может сломать код, и это обнаружится только вручную.
- **Решение**: Добавить post-execution health check: curl на страницу, проверка 404 в nginx logs, базовый Playwright smoke test.

## HIGH — Снижают эффективность

### 3. Нет feedback loop между агентами
- **Проблема**: Agent 2 слепо выполняет план Agent 1. Если план плохой — результат плохой.
- **Последствие**: Потраченные токены на выполнение заведомо неверного плана.
- **Решение**: Добавить human-in-the-loop checkpoint после PLAN.md (человек одобряет план до запуска Agent 2).

### 4. RAG context может быть stale
- **Проблема**: RAG_BASE_CONTEXT.json генерируется один раз Agent 1. Если код меняется во время работы Agent 2, RAG ответы устаревают.
- **Последствие**: Agent 2 может принимать решения на основе устаревшей структуры кода.
- **Решение**: Перегенерировать RAG base context перед каждым этапом execution.

### 5. Нет rollback механизма
- **Проблема**: Если Agent 2 сломал код, нет автоматического отката.
- **Последствие**: Ручное восстановление через git, потеря времени.
- **Решение**: Перед запуском Agent 2 делать git stash / git branch backup-<run-id>. Автоматический revert при CHANGES_REQUESTED.

### 6. SSH ControlMaster нестабилен
- **Проблема**: При частых запусках ControlSocket переполняется или ломается.
- **Последствие**: SSH команды зависают, агенты не запускаются.
- **Решение**: Уже частично исправлено (-o ControlMaster=no), но нужен health check SSH перед каждым launch.

## MEDIUM — Улучшения качества

### 7. Нет Knowledge Graph
- **Проблема**: Нет persistent graph знаний о проекте. Каждый агент начинает с нуля или с RAG.
- **Последствие**: Повторные запросы к одним и тем же файлам, потеря контекста между сессиями.
- **Решение**: Активировать gsd-graphify. Интегрировать query в agent prompts.

### 8. Agent 2 не знает об инфраструктуре
- **Проблема**: Prompt'ы агентов фокусируются на коде, но не упоминают Docker, gateway, nginx, деплой.
- **Последствие**: Агенты не понимают, что изменений недостаточно — нужен деплой.
- **Решение**: Добавить CONTEXT.md с описанием инфраструктуры в каждый контур.

### 9. Нет мониторинга падений агентов
- **Проблема**: Если Agent 2 падает с OOM или ошибкой, нет алерта.
- **Последствие**: Пользователь узнаёт о провале только при ручной проверке.
- **Решение**: Heartbeat webhook или простой poll статуса каждые 5 минут.

### 10. Односторонний поток данных
- **Проблема**: Agent 4 -> Agent 3 -> Agent 2 -> Agent 1. Нет обратной связи.
- **Последствие**: Системные ошибки (например, плохой план) не корректируются в будущих запусках.
- **Решение**: Extract learnings в конце каждого контура, обновлять project knowledge base.

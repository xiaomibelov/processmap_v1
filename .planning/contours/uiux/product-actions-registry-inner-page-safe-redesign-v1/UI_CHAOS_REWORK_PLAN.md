## 1. Контекст

Предыдущий цикл (run `20260517T112506Z-72991`) завершился `REVIEW_PASS`, но runtime на `clearvestnic.ru:5180` показал **хаотичную вёрстку** страницы «Реестр действий с продуктом». Компоненты существуют, но их расположение нарушает UX-иерархию.

Этот план описывает **rework workflow**: перестройку layout внутренней страницы реестра с сохранением всего остального.

---

## 2. Цель rework

Обеспечить:
1. **Чёткую визуальную иерархию** на странице реестра.
2. **Горизонтальную/сеточную компоновку фильтров** вместо вертикальной.
3. **Метрики под заголовком**, а не внизу страницы.
4. **Главную таблицу как доминирующий элемент**.
5. **Компактную вторичную область** для выбора источника/сессии.
6. **Сохранение shell/header/global nav** без изменений.
7. **Сохранение Analytics Hub** как контейнера.
8. **Сохранение реального потока данных** — без фейковых метрик.

---

## 3. Правильная UX-иерархия (target)

### 3.1. Заголовочный ряд
- Заголовок: «Реестр действий с продуктом»
- Короткое описание
- Справа действия: CSV, XLSX, «Вернуться» / закрыть

### 3.2. Табы области
- «Рабочее пространство»
- «Проект»
- «Сессия»
Визуально чёткие и компактные.

### 3.3. Ряд метрик (сразу под заголовком/табами)
- Сессии
- Строк
- Полных
- Неполных
- После фильтров

### 3.4. Панель фильтров и действий
- Группированные горизонтальные/сеточные фильтры:
  - Группа
  - Товар
  - Тип
  - Этап
  - Категория
  - Роль
  - Полнота
- «Сбросить фильтры»
- «Выбрано для AI: 0 / 10»
- «AI: предложить действия»
- CSV/XLSX (если не в заголовке)

### 3.5. Предупреждающий баннер
- Показывать только при необходимости.
- Размещать над таблицей.

### 3.6. Главная таблица
- Доминирующая область страницы.
- Колонки: Продукт, Действие, Процесс / шаг, Статус.
- Компактные читаемые строки.
- Бейджи «Полная» / «Неполная».
- Чипы/теги выровнены и спокойны.
- Пагинация и выбор размера страницы внизу.
- Макс. 50 строк на страницу.

### 3.7. Блок выбора источника/сессии
- Не должен доминировать в верхней части.
- Переместить во вторичную компактную область:
  - свёрнутый/details-блок;
  - правая/нижняя панель;
  - или секция «Источники данных» под таблицей.
- Должно быть понятно, что «Открыть проект» / «Открыть сессию» открывает источник, а не строку реестра.

## 5. Разделение работы (4-agent workflow)

### Agent 1 / Planner (этот документ)
- Создать все rework-артефакты.
- Убедиться, что Worker 2 и Worker 3 независимы.
- Убедиться, что Reviewer ждёт обоих DONE.

### Agent 2 / Worker — Layout implementation
**Старт:** немедленно, независимо.  
**Фокус:** перестройка layout внутренней страницы реестра.

**Обязанности:**
- Переработать layout только внутри страницы реестра.
- Сохранить shell / header / global ProcessMap UI.
- Внедрить правильную иерархию:
  заголовок → табы области → метрики → фильтры/действия → баннер → таблица → пагинация → блок источника.
- Сделать фильтры горизонтальными / сеточными, не вертикальными.
- Переместить выбор источника/сессии во вторичную компактную область.
- Сохранить реальный поток данных.
- Обновить версию до канонической следующей (`v1.0.136`).
- Все отчёты — на русском.
- Создать маркер `WORKER_2_DONE`.

**Блокировка:** если layout невозможно изменить без нарушения shell/backend/data — создать `EXEC_PART_1_BLOCKED.md`, НЕ создавать `WORKER_2_DONE`.

### Agent 3 / Worker — UX/spec/data-safety independent lane
**Старт:** немедленно, независимо. **Не ждёт Worker 2.**  
**Фокус:** спецификация, проверка неизменности, подготовка чеклистов.

**Обязанности:**
- **Не валидировать Worker 2 и не ждать его.**
- Построить UX acceptance checklist на основе скриншота и reference-иерархии.
- Классифицировать, что **должно остаться без изменений**:
  shell / header / global nav / Analytics Hub.
- Проверить, что не должно появиться фейковых чисел / данных.
- Проверить семантику и подписи блока источника/сессии.
- Подготовить runtime review checklist для Agent 4.
- Все отчёты — на русском.
- Создать маркер `WORKER_3_DONE`.

**Блокировка:** если требования неоднозначны или обнаружены конфликты с shell/Analytics Hub — создать `EXEC_PART_2_BLOCKED.md`, НЕ создавать `WORKER_3_DONE`.

### Agent 4 / Reviewer — Final runtime visual review
**Старт:** только после появления **обоих** маркеров:
- `WORKER_2_DONE`
- `WORKER_3_DONE`

**Обязанности:**
- Свежий runtime-proof на `clearvestnic.ru:5180`.
- Проверить текущую build/version.
- Открыть Analytics → «Реестр действий».
- Визуально проверить: страница больше не хаотична.
- Проверить сохранность shell / header / global UI.
- Проверить метрики, фильтры, действия, баннер, таблицу и пагинацию — в правильной иерархии.
- Проверить, что блок источника/сессии — вторичен и понятен.
- Проверить наличие CSV/XLSX и AI-контролов.
- Проверить отсутствие console errors.
- Проверить отсутствие backend/schema/BPMN/RAG изменений.
- `REVIEW_PASS` — только если браузерный runtime проходит визуально.

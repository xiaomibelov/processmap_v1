# 16_Журнал решений

## ADR-20260430-001 — Durable BPMN save ack отделён от background session sync

status:: accepted
date:: 2026-04-30

### Контекст

Performance audit `audit/project-performance-decomposition-and-slowdown-map-v1` показал, что full `GET /api/sessions/{id}` может занимать 3.41s и возвращать 4.35 MB fresh payload. В prior post-save scenario durable `PUT /bpmn` занимал около 1.15s, но UI оставался в ожидании из-за последующего full session refetch до 6.47s.

### Решение

Для hot BPMN property save path используется `backgroundSessionRefresh`.

| Решение | Деталь |
| ------- | ------ |
| Durable ack | `PUT /api/sessions/{id}/bpmn -> 200` завершает пользовательский save |
| Local truth | `diagram_state_version` и `bpmn_xml_version` берутся из backend ack и применяются через fallback session patch |
| Background sync | `GET /api/sessions/{id}` запускается background promise и позже hydrate-ит session state |
| Error policy | background refresh failure не превращает successful durable save в failed save |
| Conflict policy | `409 Conflict` после `PUT /bpmn` не скрывается и не считается durable success |

### Последствия

> [!success] Плюсы
> Пользовательский save status больше не ждёт multi-MB full session payload после successful `PUT /bpmn`.

> [!warning] Ограничение
> Full `GET /session` payload не декомпозирован. Это отдельная performance-задача. Background remote polling тоже остаётся отдельным contour.
